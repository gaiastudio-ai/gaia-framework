GAIA pre-merge gate scaffold — the exit 1 stub

Origin: AF-2026-05-31-2 / Test13 F-19 (rationale); AF-2026-06-01-1 / Test15 D-3 (this onboarding doc).

What the stub is

/gaia-init writes a starter CI workflow at the provider-canonical path (.github/workflows/gaia-pre-merge.yml for GitHub Actions, .gitlab-ci.yml for GitLab, bitbucket-pipelines.yml for Bitbucket). The single job in that scaffold is a deliberate fail-loud stub:

echo "GAIA pre-merge gate is not yet configured for this project. Run /gaia-ci-setup to wire the runner, then commit the generated workflow." >&2 && exit 1

It exits 1 on purpose. Any PR opened against the project will fail this check until the operator runs /gaia-ci-setup to replace the stub with the real GAIA pre-merge runner.

Why fail-loud instead of skip-or-warn

  • Silent green is the worst posture. A green pipeline that runs zero checks teaches the team "CI passed" means "code is fine" — but it actually means "we never asked the question". The first time a real defect lands, the team has lost weeks of false confidence. Fail-loud means the gap is visible on PR #1 of the project.
  • The scaffold is a placeholder, not a default. GAIA cannot guess the project's stack (Node vs Java vs Python vs polyglot) at /gaia-init time — that's /gaia-ci-setup's job once brownfield.platform_stack is resolved. Failing loud forces the wiring step to actually happen.
  • The error text is self-describing. The stderr line names the next command (/gaia-ci-setup) and links here for rationale. A new contributor who opens their first PR and sees the red check knows exactly what to do.

How to clear the stub

  1. Run /gaia-ci-setup in the project root.
  2. The skill resolves brownfield.platform_stack from .gaia/config/project-config.yaml, picks the matching pre-merge template, and overwrites the stub workflow file in place.
  3. Commit the resulting workflow file. Push.
  4. The next PR runs the real GAIA pre-merge gate (lint + tests + brownfield Phase-3 scan as configured) instead of the stub.

Why this doc exists

The stub error message references this page so a contributor hitting the red check has a single click to the rationale and the fix command. Without this page, the exit 1 looks like a misconfiguration; with it, the design intent is on the record.

Companion file *.user-steps.yml

Alongside the stub workflow, generate-ci-scaffold.sh writes a gaia-pre-merge.user-steps.yml (or equivalent for the chosen provider). This companion is the operator's append-only space for project-specific CI steps that GAIA's standard runner cannot guess (custom secrets injection, deploy preview, project smoke tests). It is preserved byte-for-byte across /gaia-init regenerations (FR-RSV2-38) — once you've authored steps there, regeneration cannot clobber them.
